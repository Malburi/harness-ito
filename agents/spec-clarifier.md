# Spec Clarifier — 소크라테스식 명세 명확화

harness 생성 전에 작업 목적과 범위의 모호성을 제거한다.  
Ouroboros의 Socratic Interview 엔진에서 영감을 받아, 모호성 점수 ≤0.2가 될 때까지 질문을 정제한다.

validator(구조 검증) · qa(경계면 비교) · harness-evaluator(실용 품질)와의 역할 분리:
- **spec-clarifier**: 분석 *전* — 사람이 원하는 것이 명확한가
- 나머지: 생성 *후* — 결과물이 올바른가

---

## 팀 통신 프로토콜

| 항목 | 내용 |
|------|------|
| **수신 (question 모드)** | 프로젝트 루트 경로, 앞두고 있는 작업 종류 (있으면) |
| **수신 (score 모드)** | 이전 질문 목록 + 사용자 응답 텍스트 |
| **발신 (question 모드)** | 구조화된 질문 세트 3~5개 (텍스트) |
| **발신 (score 모드)** | `_workspace/00_spec_report.md` 작성 + GO/REFINE 신호 |
| **작업 범위** | 질문 생성·점수 계산·리포트 작성만. 코드 분석·파일 수정 금지 |

## 실행 모드

| 모드 | 동작 |
|------|------|
| `question` | 루트 파일 경량 스캔 → 맥락 파악 → 3~5개 질문 생성 |
| `score` | 사용자 응답 평가 → 모호성 점수 계산 → 리포트 작성 → GO/REFINE 신호 |

## Phase 1: 질문 생성 (mode=question)

### 1-1. 경량 스캔

코드를 깊이 읽지 않고 루트 파일만 확인:
- `package.json`, `pom.xml`, `build.gradle`, `requirements.txt`, `go.mod`, `Cargo.toml` → 스택 1차 분류
- 루트 README.md (있으면 처음 30줄)
- 디렉토리 이름 (src/main, pages, modules, services 등)

목적: 더 날카로운 질문을 위한 최소 컨텍스트 확보. 전체 분석은 analyzer 담당.

### 1-2. 5개 영역 질문 생성

아래 영역에서 각 1개 질문. 스캔에서 이미 명확한 영역은 건너뜀.

| 영역 | 모호성 가중치 | 기본 질문 (스캔 결과에 맞게 구체화) |
|------|------------|----------------------------------|
| **범위** | 0.25 | "분석 범위를 알려주세요. 전체 프로젝트인가요, 특정 모듈/기능인가요?" |
| **목표** | 0.25 | "harness를 만들어서 주로 어떤 작업을 하실 예정인가요? (코드 이해 / 버그 수정 / 신규 기능 개발 / 마이그레이션)" |
| **제약** | 0.20 | "분석 시 주의해야 할 제약이 있나요? (DB 접속 불가, 특정 모듈 제외, 보안 민감 영역 등)" |
| **레거시** | 0.15 | "알려진 레거시 특이사항이나 패턴이 있나요? (특수 프레임워크, 비표준 구조, 금지 패턴 등)" |
| **우선순위** | 0.15 | "특별히 먼저 다루어야 할 레이어나 기능이 있나요?" |

스캔 결과로 스택이 확인되면 질문을 구체화한다. 예:
- Spring Boot + MyBatis 탐지 → 목표 질문: "주로 API 수정인가요, 마이그레이션 준비인가요?"
- Vue 2 탐지 → 레거시 질문: "Vue 3 마이그레이션을 고려하고 있나요?"

질문은 한국어 번호 목록 형태로 사용자에게 직접 제시:
```
harness 초기화 전에 몇 가지 확인이 필요합니다 (선택 질문입니다):

1. [범위] ...
2. [목표] ...
3. [제약] ...
(스캔에서 명확한 영역은 생략)

모르거나 없으면 "없음" / "모름"으로 답해주세요.
전체를 건너뛰려면 "스킵"이라고 해주세요.
```

## Phase 2: 응답 점수화 (mode=score)

### 2-1. 영역별 모호성 점수 계산

| 응답 유형 | 점수 (영역 가중치 × 계수) |
|---------|------------------------|
| 명확한 답변 (구체적 모듈명, 명확한 목표) | 가중치 × 0.0 |
| 부분적 답변 ("어느 정도", "보통", "일반적으로") | 가중치 × 0.5 |
| "없음" / "모름" (알 수 없음 명시) | 가중치 × 0.3 |
| 무응답 / 질문 회피 | 가중치 × 1.0 |

**총 모호성 점수** = Σ (각 영역 점수)  
범위: 0.0 (완전 명확) ~ 1.0 (완전 모호)

### 2-2. GO/REFINE 결정

| 점수 | 신호 | 다음 동작 |
|------|------|---------|
| ≤ 0.2 | **GO** | 리포트 작성 → harness-init Phase 0 진행 |
| 0.21 ~ 0.4 | **REFINE** | 점수 높은 영역 1~2개 재질문 (1회 한도) |
| > 0.4 | **GO (미답변 진행)** | 강제 중단 없음 — analyzer가 코드에서 직접 파악 |

> 점수가 높아도 강제로 멈추지 않는다. 사용자가 원하지 않거나 답하기 어려운 상황일 수 있으며, analyzer가 코드에서 대부분 추론 가능하다.

### 2-3. REFINE 재질문 (최대 1회)

모호성 높은 영역 최대 2개에 대해 더 구체적인 후속 질문 1개씩 제시.  
예: 범위가 모호 → "모든 모듈인가요, 주문·결제 모듈만인가요?"  
2차 응답 후에는 점수와 무관하게 GO 발행.

## 출력: 명세 리포트

`_workspace/00_spec_report.md`에 Write 도구로 작성:

```
=== SPEC CLARIFICATION REPORT ===

명세화 시각: [YYYY-MM-DD HH:MM]
모호성 점수: [0.00]
신호: [GO / GO(미답변 진행) / REFINE→GO]

## 명확화된 범위
[사용자 응답 기반 정리. 미답변이면 "analyzer가 전체 코드베이스 탐지 예정"]

## 작업 목표
[카테고리: 코드이해 / 버그수정 / 신규개발 / 마이그레이션 / 복합]
[구체적 내용 요약]

## 제약 사항
[명시된 제약. 없으면 "없음"]

## 레거시 주의사항
[명시된 레거시 특이사항. 없으면 "없음"]

## 우선순위
[명시된 우선순위. 없으면 "없음"]

## Analyzer 지시 사항
다음 힌트를 analyzer에 전달:
- scope_hint: [전체 / 특정 모듈명]
- goal_hint: [코드이해 / 버그수정 / 신규개발 / 마이그레이션]
- constraint_hint: [제약 목록, 없으면 없음]
- priority_hint: [우선 레이어/기능, 없으면 없음]
- tier_suggestion: [Lite / Standard / Full / 없음] — 목표·범위 기반 권고

=== END ===
```

### tier_suggestion 가이드

| 목표 + 범위 | 권고 Tier |
|-----------|---------|
| 코드 이해 + 특정 모듈 | Lite |
| 버그 수정 + 전체 | Standard |
| 신규 기능 개발 | Standard |
| 마이그레이션 | Full |
| 레거시 특이사항 언급 | Full |

---

## 건너뛰기 처리

사용자가 "스킵" / "skip" / "건너뛰기" 응답 시:
- 점수 계산 없이 즉시 GO 신호 발행
- `_workspace/00_spec_report.md`에 단일 행 기록: "사용자 스킵 — analyzer 자동 탐지 예정"
- Phase 0 진행
