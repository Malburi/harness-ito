01 — Background
출발점은 레거시 프로젝트 유지보수와 개발이었습니다
AI를 실험한 이유는 새 기술을 써보고 싶어서라기보다, 실제 프로젝트에서 반복되는 분석·수정·전환 부담을 줄이기 위해서였습니다.
🏫

hiclass

운영 중인 교육/학습 시스템 유지보수와 기능 개선. 기존 흐름을 깨지 않고 수정 영향도를 파악하는 일이 중요했습니다.

운영레거시
📚

LMS

학습 관리 기능의 유지보수와 개발. 화면, API, DB, 배치가 얽힌 흐름을 빠르게 이해해야 했습니다.

개발업무흐름
📝

전자설문

설문 생성, 응답, 집계, 통계 쪽 유지보수. 작은 쿼리 변경도 결과 화면까지 이어지는 구조였습니다.

SQL통계
🔄

self-learning

.NET 기반 시스템을 Vue3 + FastAPI 구조로 전환하는 마이그레이션. 단순 변환보다 도메인 재해석이 핵심이었습니다.

MigrationVue3/FastAPI
💡
핵심 배경: AI가 코드를 대신 짜는지보다, 기존 시스템을 더 빨리 이해하고 안전하게 바꿀 수 있는지가 더 중요했습니다.
02 — Reality
레거시 현장에서 AI를 적용하려면 코딩보다 이해가 먼저였습니다
기능 하나를 추가하더라도 기존 업무 규칙, 화면 흐름, DB 관계, 예외 처리를 모르면 AI 결과도 위험해집니다.
🧭

업무 맥락이 코드 밖에 있음

왜 이렇게 만들었는지, 어떤 예외가 있는지는 코드만 봐서는 잘 보이지 않습니다.

🔗

수정 영향도가 여러 레이어로 퍼짐

화면, API, 서비스, DAO, SQL, 배치가 연결되어 작은 수정도 운영 장애로 이어질 수 있습니다.

🧱

레거시와 신규 스택이 공존

.NET, Java, Vue, FastAPI처럼 서로 다른 기술을 동시에 이해해야 했습니다.

📉

AI 답변 품질이 컨텍스트에 크게 의존

프롬프트를 잘 써도 프로젝트 구조를 모르면 일반론 답변에서 벗어나기 어렵습니다.

🔒

보안과 권한이 실제 제약

소스, DB 스키마, 업무 문서를 아무 데나 붙여 넣을 수 없습니다.

🧪

검증 없는 자동화는 위험

AI가 만든 코드보다, 그 코드가 기존 시스템에서 안전한지 확인하는 절차가 더 중요했습니다.

03 — Evolution
AI 적용 방식은 역할 → 프롬프트 → 컨텍스트 → 하네스로 바뀌었습니다
처음에는 질문을 잘 쓰는 문제가 핵심처럼 보였지만, 실제 프로젝트에서는 “AI가 무엇을 근거로 일하게 할 것인가”가 더 중요했습니다.
Step 1
🎭

Persona

AI에게 역할을 부여했습니다. “너는 시니어 개발자야”, “DBA처럼 봐줘” 같은 방식입니다.

  • 답변 톤과 관점은 좋아짐
  • 프로젝트 사실관계는 여전히 부족
  • 역할만으로는 검증 불가
Step 2
✍️

Prompt Engineering

질문 구조를 개선했습니다. 목적, 제약, 출력 형식, 예시를 명확히 주는 방식입니다.

  • 결과물이 더 정돈됨
  • 반복 작업에는 효과 있음
  • 긴 프로젝트 맥락은 매번 다시 설명해야 함
Step 3
📚

Context Engineering

코드, 문서, DB 정보, 업무 규칙을 함께 제공하기 시작했습니다.

  • 프로젝트 맞춤 답변 가능
  • 컨텍스트 선별이 중요해짐
  • 정보가 많아질수록 관리가 어려움
Step 4
🧩

Harness Engineering

컨텍스트를 일회성으로 주는 게 아니라, 프로젝트 지식 체계와 역할별 워크플로우로 구성했습니다.

  • 인덱스와 패턴을 재사용
  • 에이전트별 역할 분리
  • 분석-수정-검증 흐름을 표준화
04 — Tools
Claude Code, Codex, Antigravity를 써보며 각 도구의 강점을 나눠봤습니다
도구를 하나로 정답 내리기보다, 어떤 작업에 어떤 방식이 맞는지 직접 프로젝트에 적용해보는 과정이었습니다.
🧠

Claude Code

코드베이스 맥락을 길게 다루고, 대화형으로 설계/분석/수정 흐름을 이어가기에 좋았습니다.

배운 점: 프로젝트 지식을 잘 넣어주면 단순 코딩보다 분석 파트에서 강해집니다.
🛠️

Codex

로컬 워크스페이스에서 파일을 읽고 고치고 검증하는 개발 에이전트 방식에 적합했습니다.

배운 점: 수정 자체보다 변경 범위를 좁히고 검증하는 절차가 중요합니다.
🚦

Antigravity

AI 기반 개발 흐름과 작업 자동화 관점에서, 기존 IDE와 다른 방식의 가능성을 확인했습니다.

배운 점: 도구가 달라도 결국 필요한 것은 안정적인 컨텍스트와 검증 기준입니다.
⚠️
정리: AI 도구의 차이보다 더 큰 차이는 “프로젝트 지식을 어떻게 구조화해서 AI에게 줄 것인가”였습니다.
05 — Migration Case
self-learning 마이그레이션에서 하네스의 필요성이 더 분명해졌습니다
.NET에서 Vue3 + FastAPI로 옮기는 일은 문법 변환이 아니라, 기존 업무 규칙을 새 구조에 안전하게 재배치하는 일이었습니다.

AS-IS

.NET 기반

기존 화면, 서버 로직, DB 접근, 예외 처리가 한 시스템 안에 강하게 묶여 있었습니다.

화면과 서버 흐름이 기존 구조에 결합
업무 규칙이 코드 곳곳에 분산
테스트보다 운영 경험에 의존
변경 영향도 파악이 느림
→

TO-BE

Vue3 + FastAPI

프론트와 API를 나누면서도 기존 업무 규칙, 데이터 계약, 예외 처리를 잃지 않아야 했습니다.

화면 컴포넌트와 API 계약 분리
도메인 규칙을 서비스 레이어로 재정리
마이그레이션 단위와 우선순위 설계
기존 기능 동등성 검증 필요
💡
여기서 얻은 결론: AI에게 “이걸 Vue3/FastAPI로 바꿔줘”라고 묻는 것보다, 기존 시스템의 규칙·계약·리스크를 먼저 구조화하는 게 더 중요했습니다.
06 — Harness
그래서 현재는 HD현대 Claude Code 기반 Harness Engineering을 만들고 있습니다
하네스는 AI에게 매번 설명하는 자료가 아니라, 프로젝트를 이해하고 작업하기 위한 지속 가능한 작업 환경입니다.

🧩 하네스가 담는 것

코드 구조
모듈, 엔트리포인트, 호출 그래프, 주요 클래스/메서드
DB/SQL
SQL 사용처, 테이블 관계, 트랜잭션 경계, 성능 위험 후보
업무 패턴
Controller-Service-DAO, 화면 흐름, 배치, 예외 처리 컨벤션
레거시 지식
오래된 코드의 목적, 숨은 규칙, 위험 구간, 마이그레이션 후보
검증 기준
변경 전 확인할 영향도, 테스트 후보, 롤백 포인트

🤖 하네스 위에서 하는 일

영향도 분석
“이 메서드 바꾸면 어디까지 영향이 가나?”를 직접/간접 경로로 확인
안전한 수정
변경 전 체크 → 코드 수정 → 검증 → 문서 반영 흐름으로 작업
SQL 리뷰
N+1, 인덱스, 풀스캔, 권한, 트랜잭션 위험을 사전 점검
스캐폴딩
기존 프로젝트 컨벤션에 맞춰 기능 뼈대 생성
마이그레이션 계획
전환 범위, 우선순위, 난이도, 리스크를 근거 기반으로 정리
07 — Shift
핵심 변화는 AI에게 일을 시키는 방식이 바뀐 것입니다
프롬프트만 잘 쓰는 단계에서, 프로젝트 지식과 작업 절차를 갖춘 AI 협업 환경을 만드는 단계로 넘어왔습니다.
이전 방식: 일회성 AI 사용
  • ❌
    매번 코드 일부를 붙여 넣고 같은 배경 설명을 반복합니다.
  • ❌
    AI가 프로젝트 전체 구조를 모르기 때문에 일반론 답변이 많습니다.
  • ❌
    좋은 답변이 나와도 다음 작업에서 재사용하기 어렵습니다.
  • ❌
    수정 결과를 검증하는 기준이 사람마다 달라집니다.
  • ❌
    보안, 권한, 컨텍스트 관리가 체계화되지 않습니다.
현재 방향: Harness Engineering
  • ✅
    프로젝트 지식을 인덱스와 패턴으로 만들어 반복 활용합니다.
  • ✅
    AI가 호출 그래프, SQL, 트랜잭션 등 근거를 보고 답합니다.
  • ✅
    역할별 에이전트와 워크플로우로 작업 품질을 일정하게 만듭니다.
  • ✅
    분석, 수정, 검증, 문서화를 하나의 절차로 묶습니다.
  • ✅
    사내 보안 정책에 맞게 컨텍스트 공유 범위를 통제합니다.
08 — What I Want To Share
제가 공유하고 싶은 건 AI 도구 소개보다 적용 방식입니다
AI를 잘 쓰는 문제는 결국 우리 프로젝트를 얼마나 잘 설명하고, 안전하게 일하게 만들 수 있느냐의 문제였습니다.
🧠

AI는 맥락 없이는 평범합니다

좋은 모델을 써도 프로젝트 구조, 업무 규칙, 제약 조건을 모르면 답변은 일반론에 머뭅니다.

📚

컨텍스트는 자료가 아니라 설계 대상입니다

무엇을 AI에게 줄지, 어떤 형식으로 유지할지, 누가 검증할지 정해야 합니다.

🛡️

자동화보다 안전한 협업이 먼저입니다

AI가 만든 결과를 사람이 판단할 수 있게 근거, 영향도, 검증 절차를 같이 제공해야 합니다.

✅
정리: AI를 프로젝트에 적용한다는 것은 “프롬프트를 잘 쓰는 것”에서 끝나지 않고, 프로젝트 지식과 작업 절차를 엔지니어링하는 일로 확장됩니다.
